# spec.md 完整模板

## 基础模板

```markdown
# 任务：{一句话描述}

> Task ID: {task_id}
> 引擎: {omc|omx} | 工作目录: {cwd}
> 创建时间: {timestamp}

## 目标
{核心目标，一段话说清楚要达成什么}

## 项目上下文
- **项目路径**: {绝对路径}
- **技术栈**: {语言/框架}
- **当前分支**: {git branch}
- **项目类型**: {web应用/CLI工具/库/服务等}

## 详细需求
{完整需求描述，保持用户原文，不做删减}

## 约束条件
- {必须遵守的约束1}
- {必须遵守的约束2}
- 保持代码风格一致
- 不破坏现有功能
- 遵循项目约定

## 验收标准
- [ ] {可验证的标准1}
- [ ] {可验证的标准2}
- [ ] {可验证的标准3}
- [ ] 所有测试通过
- [ ] 代码审查通过
```

## 生成规则

### 内容提取
1. **任务描述**：从用户消息第一句提取，去掉冗余词汇
2. **核心目标**：重新表述用户需求，明确要达成的结果
3. **详细需求**：保持用户原文，包含所有细节和上下文

### 路径处理
4. **项目路径**：必须使用 Step 0 预检通过的绝对路径
5. **禁止简写**：不能使用 `~`、`./` 等相对路径
6. **路径验证**：spec.md 中的路径必须确保存在

### 约束推断
7. **技术栈约束**：根据项目文件推断（package.json→Node.js，requirements.txt→Python等）
8. **风格约束**：检查现有代码，保持一致性
9. **兼容性约束**：不破坏现有 API 或接口

### 验收标准
10. **可验证性**：每个标准都必须可以通过命令或检查验证
11. **具体性**：避免主观描述，使用客观指标
12. **完整性**：覆盖功能、性能、质量等维度

## 示例场景

### Web 应用功能开发
```markdown
## 项目上下文
- **项目路径**: /Users/qiuming/projects/dashboard-app
- **技术栈**: React 18 + TypeScript + Vite + Tailwind CSS
- **当前分支**: feature/user-auth
- **项目类型**: SPA 用户管理后台

## 约束条件
- 遵循现有 TypeScript strict 模式
- 使用项目统一的 API client（axios）
- UI 组件必须支持 dark mode
- 遵循 React Hooks 最佳实践
```

### API 服务开发
```markdown
## 项目上下文
- **项目路径**: /Users/qiuming/services/user-api
- **技术栈**: FastAPI + SQLAlchemy + PostgreSQL + Redis
- **当前分支**: main
- **项目类型**: REST API 微服务

## 约束条件
- 遵循 OpenAPI 3.0 规范
- 所有端点必须有认证
- 数据库操作必须事务化
- 错误响应使用统一格式
```

### CLI 工具开发
```markdown
## 项目上下文
- **项目路径**: /Users/qiuming/.hermes/tools/config-manager
- **技术栈**: Python 3.9 + Click + YAML + JSON
- **当前分支**: main  
- **项目类型**: 命令行配置管理工具

## 约束条件
- 兼容 Python 3.8+
- 配置文件向后兼容
- 错误信息用户友好
- 支持 --help 和 --version
```

## 复杂任务扩展

### 审计任务 spec
审计类任务需要在 spec 中预嵌入数据：

```markdown
## 审计范围
- **文件范围**: src/ 目录下 42 个 .py 文件
- **代码行数**: 总计 15,847 行（不含注释和空行）
- **已知问题**: 3 个安全漏洞，5 个性能问题

## 预收集数据
{Hermes 预先收集的统计信息、分布数据、边界情况}

## 审计维度
1. **安全性**：SQL 注入、XSS、认证绕过
2. **性能**：N+1 查询、内存泄漏、并发瓶颈
3. **可维护性**：代码重复、复杂度、测试覆盖率
```

### 重构任务 spec
重构类任务必须包含完整上下文：

```markdown
## 重构目标
从 {当前架构} 重构到 {目标架构}

## 现状分析
{从审计报告提取的问题清单，按优先级排序}

## 修复方案
{具体的修改策略，包含文件、函数、类级别的变更}

## 风险控制
- 增量迁移，保持功能不变
- 每个模块独立验证
- 回滚策略
```
